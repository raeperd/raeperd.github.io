# Write tests Not too many Mostly integration
## Write tests 
- Of course
- Enhanced Workflows
- Confidence

## Not too many
- Code coverage 
	- At some points higher coverage does not means higher performance
	- There is some sweet spots between coverage and productivity

### Implementation Details 
- If refactor breaks test, it is implementation details
- If test does something consumer doesn't then it's implementation details 

## Mostly Integration 
- Unit test does not any good if you don't **also** verify integration 
- Stop mocking so much stuff

> Knowing what to test gives true confidence
